## Context

The current implementation delivers a functional multi-service prototype with GPS ingest, emergency alerts, student presence processing, compliance, video, student management, a working admin dashboard, a working parent portal, and a driver app with offline buffering.

The emergency alerts service already exposes both WebSocket broadcast and an SSE stream. The parent app, however, still polls for alerts and does not subscribe to SSE. There is no notification consumer pipeline that processes BullMQ jobs published by emergency-alerts and student-presence. Phase 1 closes this gap by delivering the complete parent safety communication loop.

## Scope

### 1. Notification Delivery Backbone

- Create a proper notification service boundary or equivalent dedicated notification module with clear consumer responsibilities.
- Consume emergency alert and presence jobs from BullMQ instead of leaving events unhandled after publication.
- Introduce provider abstraction for push, SMS, and email, even if only one delivery channel is enabled initially.
- Persist notification attempts and statuses for audit and support.

### 2. Parent Delivery Workflows

- Deliver emergency alerts to parents assigned to the impacted route.
- Deliver boarding and alighting notifications using presence events.
- Add a parent-facing notification list or recent activity feed.

### 3. Parent App Real-Time Integration

- Replace alert polling with the existing SSE stream where appropriate.
- Preserve polling only as a fallback strategy, not the primary design.
- Add UI states for unread, failed, or delayed notifications.

## Dependencies

- Existing BullMQ producers in emergency-alerts and student-presence services.
- Parent-to-route mapping in seeded and operational data.
- No upstream phase dependency — this is the first phase.

## Acceptance Criteria

- An emergency alert generated by the driver is delivered to the correct parent audience.
- A presence event can trigger a parent-visible notification without manual intervention.
- Delivery attempts are queryable for debugging and audit.
- Parent app can receive live alert changes without relying solely on periodic polling.

## Verification

- Integration tests for queue producer-to-consumer flow.
- End-to-end tests covering alert and presence notifications.
- Demo validation using seeded parent accounts and live route data.

## Demo Impact

After this phase, the demo can show real parent alerts instead of narrated future-state behavior.

## Estimated Complexity

High. Requires introducing a new service boundary or module, wiring two separate BullMQ consumers, implementing provider abstraction, persisting delivery state, and updating the parent app to use SSE.
